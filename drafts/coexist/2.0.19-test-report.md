## Bugs found

### Bug 1 - "built by Ecodia" mark does not render in Spectral (LOW, cosmetic) - **FIXED 2026-07-13 in `1d900f7`, verified on the deployed URL. See "Final gate" below.**
`EcodiaAttribution` declares `fontFamily: 'Spectral', 'Iowan Old Style', Garamond, 'Times New Roman', serif`, but **no Spectral webfont is bundled**: there is no `@font-face` for it in source CSS, none in the built `dist` CSS, and no font file in the repo. The only registered faces on the live page are Montserrat and Eau Sans.

Measured on the live app (canvas width of "built by Ecodia" at 15px):
```
Spectral        91.65   <- identical to a bogus family => not available, silently falls back
ZzzNotARealFont 91.65
Times New Roman 91.65   <- Chrome's default fallback
Iowan Old Style 99.92   <- a genuinely present face measures differently (probe discriminates)
```
So the mark renders in a fallback serif on every surface. The *upright* half of the 2026-07-09 change is correct (no italic class). This is exactly the failure shape in `declared-custom-fonts-must-be-bundled-and-render-verified-2026-06-18`.

Fix: bundle a Spectral face (self-hosted `@font-face`, so it also works offline in the native webviews) rather than naming it in a font stack.

**FIXED 2026-07-13 (`1d900f7`).** Spectral regular 400 and italic 400 are bundled as woff2 with a real `@font-face`. Re-probed against the deployed app: Spectral now measures **99.88** while the bogus family and Times both stay at 91.65, and `document.fonts` reports both faces loaded. Both files are inside the APK too, so the native webview has the font offline. The stale "Spectral italic signature" docstring is corrected; the mark is upright roman.

### Bug 2 - the 2.0.19 headline fix had no test guarding it (LOW) - FIXED
`deriveStatus` seats a paid-but-waitlisted person as Going by checking the confirmed ticket **before** the waitlisted registration. Nothing pinned that ordering. The suite covered waitlisted-*without*-a-ticket and confirmed-*with*-a-registration, but never waitlisted-**with**-a-confirmed-ticket, which is the actual Kieren/Angelica case that 5b23d53 fixed.

I proved the gap with a negative control: with the two branches deliberately swapped (bug reinstated), **my new test fails while all 16 pre-existing tests still pass**. Reverted, test added, pushed to main as `3b07de7`. Suite now 409/409.

### Bug 3 - required-answer validation accepts an explicit JSON `null` (LOW, not user-reachable)
`validate_ticket_answers` catches a *missing* key (SQL NULL), an empty string, and an empty array, but a key present with JSON `null` has `jsonb_typeof = 'null'` and slips through, so a crafted API call could skip a required question. The client cannot produce this (the modal's `isBlank()` treats null as blank and keeps Continue disabled), so it is defence-in-depth only. Fix would be adding `jsonb_typeof(v) = 'null'` to the guard.

### Bug 4 - local `npx eslint .` reports 691 phantom errors (dev ergonomics, not a product bug)
The repo has 20 `_worktrees/*` checkouts that are **not git-ignored and not eslint-ignored**, so the CI lint command run locally lints all of them. CI does a clean checkout and never sees them. Excluding `_worktrees`, eslint is **0 errors / 200 warnings / exit 0**. This cost me real time chasing a phantom red gate and will cost the next person the same. Fix: add `_worktrees/**` to the eslint ignores and `.gitignore`.

### Why it had never completed, in three prior attempts

The cause was infrastructure. The emulator was being **killed by an AVD lock collision**:

```
FATAL | Running multiple emulators with the same AVD is an experimental feature.
        Please use -read-only flag to enable this feature.
```

Booting a second emulator on an AVD that is already running tears down the one **already running**. A sibling agent was gating the same app on this host and kept booting `coexist_aosp`. The victim died mid-suite, and every flow after that failed with `device 'emulator-5556' not found`, so an infrastructure collision presented as a wall of app regressions. One run died at flow **42 of 65 after 41 green flows**, which is exactly the shape that got read as "the host was thrashing".

Falsified along the way, so nobody re-chases them: **memory** (64GB box, 70% free), **the GPU** (died identically under `swiftshader_indirect`, `host` and `guest`), and **a crash** (no qemu crash report; the emulator was asked to shut down).

Fix: `-read-only` in `backend/scripts/app-tests/run-app-tests.sh`, which lets same-AVD emulators coexist. The suite then ran 65/65 without losing the device.

### The 3 reproducible failures, all harness bugs

**05-admin-metrics-invariance.** Two separate defects, neither in the app.
1. The prehook exported a bare `1000` while the stat card runs the value through `toLocaleString('en-AU')` and paints `1,000`. The assertion failed on the comma. Fixed: the prehook now emits the formatted form.
2. The deeper one, and a real (pre-existing, admin-only) product defect: the REGISTRATIONS stat is **capped at 1,000 by PostgREST's row limit**. It counts *fetched rows*, and PostgREST caps a response at 1000. True count over the app's own 200-upcoming + 200-past window is **1543**, so the card under-reports by 35%. The invariance test could never catch it because the prehook mirrors the app's fetch-and-count and hit the same cap, so test and app agreed on the same wrong number. A test that mirrors the implementation cannot catch a bug in the implementation. Filed, status_board `86ca5dc3`. **Not a 2.0.19 regression** and not a store blocker.

**09-notifications-walk.** Asserted one of `unread notification|All caught up|No notifications`. Those are the **empty-state** strings. The screenshot shows a healthy populated screen: 3 notifications, date-grouped, with working "Mark read" swipe actions and no summary banner, so a correct populated screen could never satisfy the assertion. Fixed (`871ceaf`).

**99-walkin-count-verify.** Asserted the first stat card reads `Registered` / `61`. The card is now `Going` / `84`, and 84 is correct. DB probe: 61 registrations (registered|attended), 23 walk-ins, 30 attended, 0 waitlisted. Going counts registrations plus walk-ins (61+23=**84**); Checked In counts attended plus walk-ins (30+23=**53**). The invariant Going >= Checked In holds. The flow's own load-bearing probe (Checked In == 53) passed throughout. Fixed (`871ceaf`).

Both fixed flows re-run green.
