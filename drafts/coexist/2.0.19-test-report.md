`code@ecodia.au` already has `profiles.role = 'admin'`.

The matrix confused two different axes: `collective_members.role` (which is `participant` on Sunshine Coast) and the global `profiles.role`. The gate functions read the latter:

- `is_admin_or_staff(uid)` = `role IN ('national_leader','manager','admin')` -> **true**
- `events_insert_leader` policy = `is_collective_leader_or_above(...) OR is_admin_or_staff(...)` -> **true**
- `ticket_questions_manage` policy = event creator `OR role IN ('admin','national_leader')` -> **true**

**No DB role grant was made.** The leader flows were drivable as-is.

### Row 2 + 4 + 8 + 12: the headline feature, proven end to end

This is the part that matters most, and it is proven with discriminating probes.

**A real production event is already using the feature.** "Wild Mountains Conservation Campout" (`99d2b098`) carries 4 active custom questions and 55 tickets.

**Zero tickets anywhere in the database carry `custom_answers`.** I chased this as a possible persistence bug and falsified it: the only 2 tickets created *after* the questions existed share an identical timestamp across two different users, which is a bulk admin **grant** (`grant-event-ticket`), not a buyer checkout. So the buyer path had simply never been exercised in production. That meant I had to exercise it myself rather than trust prod data.

**Required-answer gate (server side).** Reserving with no answers is refused:
```
reserve_event_ticket(..., p_answers => NULL)
-> ERROR 23514: Missing required answer: Will you be arriving by four-wheel-drive (4WD)?
   at validate_ticket_answers() line 8
```

**Persistence + export, in one rolled-back transaction (zero production mutation).** Reserving *with* answers persisted them exactly, and the export RPC surfaced them on a confirmed ticket:
```json
{ "email": "code@ecodia.au", "ticket_status": "confirmed", "registration_status": "registered",
  "custom_answers": { "<4wd>": true, "<prev-attended>": false,
                      "<travelling-from>": "Brisbane QLD", "<consent>": true } }
```
Booleans survive as booleans (including `false`), text survives, and the confirmed ticket reads as **Going** even though a separate registration row exists. That single probe covers rows 2, 4, 8 and 12 at the data layer.

**Then the same thing through the actual browser** (canonical CDP, mobile viewport 390x844), on a private $0 test event I created and deleted:

- The resulting ticket row carried exactly what I typed: `custom_answers = {"<badge>": "QA Regression Badge", "<vehicle>": true}`

## Bugs found

### Bug 1 - "built by Ecodia" mark does not render in Spectral (LOW, cosmetic, not a blocker)
`EcodiaAttribution` declares `fontFamily: 'Spectral', 'Iowan Old Style', Garamond, 'Times New Roman', serif`, but **no Spectral webfont is bundled**: there is no `@font-face` for it in source CSS, none in the built `dist` CSS, and no font file in the repo. The only registered faces on the live page are Montserrat and Eau Sans.

Measured on the live app (canvas width of "built by Ecodia" at 15px):
```
Spectral        91.65   <- identical to a bogus family => not available, silently falls back
ZzzNotARealFont 91.65
Times New Roman 91.65   <- Chrome's default fallback
Iowan Old Style 99.92   <- a genuinely present face measures differently (probe discriminates)
```
So the mark renders in a fallback serif on every surface. The *upright* half of the 2026-07-09 change is correct (no italic class). This is exactly the failure shape in `declared-custom-fonts-must-be-bundled-and-render-verified-2026-06-18`.

Fix: bundle a Spectral face (self-hosted `@font-face`, so it also works offline in the native webviews) rather than naming it in a font stack. I did **not** ship this: it is a brand-asset decision (which weights, self-host vs CDN, bundle-size cost) and it is not a store blocker.

Minor: the component docstring still says "Spectral **italic** signature" though the mark is now correctly upright.

### Bug 2 - the 2.0.19 headline fix had no test guarding it (LOW) - FIXED
`deriveStatus` seats a paid-but-waitlisted person as Going by checking the confirmed ticket **before** the waitlisted registration. Nothing pinned that ordering. The suite covered waitlisted-*without*-a-ticket and confirmed-*with*-a-registration, but never waitlisted-**with**-a-confirmed-ticket, which is the actual Kieren/Angelica case that 5b23d53 fixed.

I proved the gap with a negative control: with the two branches deliberately swapped (bug reinstated), **my new test fails while all 16 pre-existing tests still pass**. Reverted, test added, pushed to main as `3b07de7`. Suite now 409/409.

### Bug 3 - required-answer validation accepts an explicit JSON `null` (LOW, not user-reachable)
`validate_ticket_answers` catches a *missing* key (SQL NULL), an empty string, and an empty array, but a key present with JSON `null` has `jsonb_typeof = 'null'` and slips through, so a crafted API call could skip a required question. The client cannot produce this (the modal's `isBlank()` treats null as blank and keeps Continue disabled), so it is defence-in-depth only. Fix would be adding `jsonb_typeof(v) = 'null'` to the guard.

### Bug 4 - local `npx eslint .` reports 691 phantom errors (dev ergonomics, not a product bug)
The repo has 20 `_worktrees/*` checkouts that are **not git-ignored and not eslint-ignored**, so the CI lint command run locally lints all of them. CI does a clean checkout and never sees them. Excluding `_worktrees`, eslint is **0 errors / 200 warnings / exit 0**. This cost me real time chasing a phantom red gate and will cost the next person the same. Fix: add `_worktrees/**` to the eslint ignores and `.gitignore`.

## Maestro suite

**The full 65-flow suite did not complete.** I am reporting that straight, without a fabricated X/65.

Cause: host resource exhaustion. Load average hit **22-27** (WindowServer 46%, BetterDisplay 44%), which starved the emulator until `adb` went unresponsive and Maestro lost its socket:
```
java.io.IOException: Command failed (tcp:60776): closed
  at dadb.adbserver.AdbServer.send$dadb(AdbServer.kt:103)
```
Those runs are noise. They say nothing about the app. I made three attempts, including killing everything else and rebooting the AVD clean.

**Focused 6-flow subset on a healthy, uncontended emulator: 5/6 pass.**

| Flow | Result |
|---|---|
| 01-signin-authed-feed | pass |
| 04-settings-walk | pass |
| 07-profile-tickets | pass |
| 37-events-detail-render | pass |
| 90-cold-start-render | FAIL -> **disproven, see below** |
| 99-auth-signed-out-sweep | pass |

Three separate failures were investigated and **all three turned out to be harness artefacts**:

1. `01/02/03` failing on `"YOUR NEXT EVENT" is visible` - the test account had **no active upcoming registration** (its only one was `cancelled`, almost certainly by a prior cleanup flow). The app was rendering "No events on your calendar yet", which is *correct*. I restored the fixture and 01 then passed.
2. `02-tabs-walk` failing on `"Type a message"` - the composer **is clearly visible in the pixels**; the Capacitor WebView a11y tree just did not expose the placeholder as a matchable node (`capacitor-webview-a11y-tree-fills-after-pixels`).
3. `90-cold-start-render` failing with a blank screenshot - I probed this directly because a genuine cold-start blank *would* be a store blocker. After a full `pm clear` (stronger than Maestro's `clearState`), **the login screen renders correctly at 15s**, with no JS error, no `AndroidRuntime` crash. The blank frame was a mid-paint capture under emulator load.

## What did not get done (and what I would do next)

1. **Run the full 65-flow suite on a quiet host.** This is the one thing standing between "GO on the evidence gathered" and "GO on the full gate". It needs a machine that is not simultaneously running WindowServer/BetterDisplay at ~90% combined.
2. Rows 9 (native hero overscroll), 3 (guest checkout UI), 5 (campout dietary/medical modal UI), and the create-event questions-editor UI in row 1 were not driven. None of them is a code path I found reason to doubt; they are simply unobserved.
3. Row 13's real question - *do Sentry events actually arrive in the Co-Exist project* - is unanswered, and there is a standing P1 that Sentry canaries are dead on 5 sibling apps. Worth a canary before leaning on Sentry for this release.
